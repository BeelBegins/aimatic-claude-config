## Reverse proxy: Caddy + nginx (recurring HTTPS-header gotcha)

Request path for every site: **Caddy** (`/etc/caddy/Caddyfile`, public TLS termination, one `reverse_proxy` block per site → `127.0.0.1:<port>`) → this bench's own **nginx** (`config/nginx.conf`, symlinked from `/etc/nginx/conf.d/frappe-bench.conf` — editing the repo file *is* editing the live config) → gunicorn (`127.0.0.1:8000`). Ports: szl=8080, siezal=8081, hsm=8082 (one more per site added going forward).

Each site's `location @webserver` block **must** set `proxy_set_header X-Forwarded-Proto $http_x_forwarded_proto;` (passing through the value Caddy already set correctly) — **not** the nginx-generated default `proxy_set_header X-Forwarded-Proto $scheme;`, which reflects only the Caddy→nginx hop (always plain HTTP) and overwrites Caddy's correct value with a wrong "http". This breaks any HTTPS-only server-side check, notably `aimatic.offline_pos.api._is_https_request()` / `_require_https_for_pos_admin_authorization()` — the Posapp Electron client's **administrator/Settings login** (`authorize_pos_admin_action`) requires HTTPS and throws "HTTPS is required for supervisor authorization" when this is wrong, while **cashier login** (`pos_cashier_login`) has no such check and keeps working fine. That specific symptom — cashier login fine, administrator login broken — is the signature of this exact bug.

**This fix does not persist on its own.** `config/nginx.conf` is regenerated by bench (e.g. `bench setup nginx`, which runs whenever a new site is added — hit this in practice when `hsm` was added and it silently reverted all three sites' blocks back to `$scheme`) or by any other tool/agent that touches nginx config. After any site-add or nginx-config change, grep for `X-Forwarded-Proto \$scheme` in `config/nginx.conf` and re-apply the `$http_x_forwarded_proto` fix to every server block if found. Reloading requires a real privileged terminal (`sudo nginx -t && sudo systemctl reload nginx`) — this cannot be done from a sandboxed/non-TTY session; ask the user to run it themselves.

## Legacy iPOS data migration

Whenever migration/import from the old iPOS software comes up (items, suppliers, or a new site's
cutover), the entire toolkit — mapping docs **and** the executable per-site scripts — lives in one
place: **`apps/aimatic/ipos_data_migration/`**. This is a deliberate consolidation (2026-07-14):
scripts used to be written ad hoc into each site's `sites/<site>/private/files/`, which is not
git-tracked and scatters the migration knowledge across sites instead of keeping it in the one repo
actually owned by this project. Do not go back to that pattern — a script's `FILE_PATH` constant
points at wherever the source workbook was actually uploaded on that site (`sites/<site>/public/
files/...` or `private/files/...`), but the script file itself belongs in this directory, not there.

- `apps/aimatic/ipos_data_migration/import.md` — canonical runbook for old-software **item**
  imports (barcodes, pricing, FBR category, stock, Category/Item Group mapping) on `szl`, `siezal`,
  `hsm`, and future sites using the same workbook shape.
- `apps/aimatic/ipos_data_migration/supplierimport.md` — canonical runbook for legacy **supplier**
  imports (separate from items) — NTN-based dedup/merge rules, opening-balance posting mechanism,
  Filers/Non-Filers Tax Withholding Group scheme, Contact mapping — driven off a legacy
  `schema.xlsx` shape (`SupplierCode, SupplierName, FBRTYPE, ContactPerson, [ContactNumber,]
  NTNo, StandardNTN, WhtTax%, LedgerCode, TotalDebit, TotalCredit, ClosingBalance`).
- `import_hsmitems.py` / `import_hsm_suppliers.py` and `import_siezal_items.py` /
  `import_siezal_suppliers.py` — per-site reference scripts, same directory. The `siezal` pair is
  the more complete starting point for a new site (adds the Item Group tree, `MRP` fallback chain,
  `Exempt` WHT-category naming, Contact-record mapping — see the two `.md` files for what's specific
  to `siezal`'s data vs. generally reusable).
- Do not keep the authoritative mapping or a script only inside a site-private file such as
  `sites/<site>/private/files/import.md`; site-private copies can exist as run artifacts, but the
  maintained source of truth belongs in `apps/aimatic/ipos_data_migration/`.
- When import behavior changes, or a new site's import is built, update `import.md` (items) and/or
  `supplierimport.md` (suppliers) in the same session, and add the new site's scripts to this same
  directory (copy the most complete existing script rather than writing from scratch). `import.md`
  captures field mapping decisions and operational prerequisites discovered during item imports,
  including:
  - `ItemCode` and `RefCode` being treated as the first and second plain barcodes
  - `Pcs` needing `Must be Whole Number = No` when source `Onhand` contains decimals
  - stock-entry rows needing zero valuation allowed when source `CurCost = 0`
- `supplierimport.md` captures the supplier-side equivalents, most notably: group legacy rows by `StandardNTN` (not raw `NTNo`) since `Supplier.tax_id` must be unique; within a duplicate-NTN group the highest-activity row (`|TotalDebit| + |TotalCredit|`) decides the merged Supplier's name/FBRTYPE/WhtTax%, but *all* legacy `SupplierCode`s are kept (comma-joined in `Supplier.custom_legacy_supplier_code`), not just the winner's; opening balances post as one Journal Entry **per legacy row** (not per merged Supplier) so a merged Supplier's ledger still shows which original vendor/brand contributed each amount.

## `aimatic` app architecture

Hooks are declared in `apps/aimatic/aimatic/hooks.py`. Key wiring to know before changing behavior:

- `doc_events`: `POS Invoice` → `aimatic.fbr_pos.events` (validate/before_submit), plus `on_submit`/`on_cancel` shared with `loyalty`/`gift_voucher` (see those modules below); `Customer` → `aimatic.offline_pos.customer_validation.validate_customer`; `Sales Order`/`Sales Invoice`/`Delivery Note`/`Purchase Order`/`Purchase Invoice`/`Purchase Receipt`/`Stock Entry` → `aimatic.branch_management.events.apply_branch_defaults` (**`before_validate`**, deliberately not `validate` — see the hook-ordering note below); `Purchase Invoice`/`Purchase Receipt` also → `aimatic.item_pricing.events.update_latest_price_incl_taxes` (on_submit); `Purchase Receipt` also → `aimatic.shelf_pricing.events.validate_shelf_price_before_submit` (before_submit) and `aimatic.shelf_pricing.events.restore_prices_on_cancel` (on_cancel); `Purchase Receipt` also → `aimatic.purchase_history_autofill.events.autofill_purchase_receipt_item_fields` (before_validate, after the other two Purchase Receipt before_validate hooks — see the `purchase_history_autofill` module below); `Branch` → `aimatic.branch_management.events.validate_branch_company_consistency` (validate); `POS Profile` → `aimatic.branch_management.events.validate_pos_profile_cost_center` (validate).
- **Hook ordering gotcha, learned the hard way**: for the `"validate"` doc_event specifically, Frappe's hook composer (`Document.hook`'s `compose`/`runner` in `frappe/model/document.py`) calls the DocType controller's *own* `validate()` method **first**, then iterates app-registered `doc_events["validate"]` hooks **afterward** — and if the controller's own `validate()` raises, the hooks loop never runs at all. This bit us concretely: `apply_branch_defaults` on `"validate"` never got a chance to fix a row's `cost_center`/`warehouse`/`rejected_warehouse` before ERPNext's own core `Purchase Receipt` validation (`SubcontractingController.validate_rejected_warehouse`) already threw "Accepted and Rejected Warehouse cannot be same" using not-yet-corrected values. Moving the hook to `"before_validate"` (a distinct, strictly-earlier lifecycle event — confirmed via `frappe/model/document.py`'s `_validate()`, which calls `run_method("before_validate")` before `run_method("validate")`) fixed it. Rule of thumb: if a hook needs to run *before* a doctype's own core validation logic can object to something, it must be `before_validate`, not `validate`.
- `doctype_js`: `Purchase Receipt` gets two client-only scripts (a list value, both supported by Frappe's `get_code_files_via_hooks`): `public/js/purchase_receipt_label_printing.js` (adds a "Create Barcode Labels" button) and `public/js/purchase_receipt_history_autofill.js` (live pre-save preview for `purchase_history_autofill`, see below); `Delivery Note`/`Stock Entry`/`Sales Invoice` share `public/js/label_printing_source_buttons.js` for the same button (Stock Entry only for submitted Material Transfer entries); `Supplier` gets `public/js/supplier_vendor_performance.js`, which adds a `Vendor Performance` button that opens the vendor-performance page with that supplier preselected — none of these doctypes are ever modified server-side.
- `fixtures`: Custom Field / Property Setter / Client Script / Server Script are fixture-tracked by an explicit name **exclude**-list in `fixture_exclusions.json` (2026-07-15; was previously an allowlist requiring manual maintenance for every new customization — flipped specifically so new work is captured automatically). Every record of these 4 doctypes is included as a fixture *except* the ones named there — Frappe's own Accounting Dimension auto-injected `-branch` fields (recreated automatically by Frappe whenever the dimension is applied, not something that needs hand-tracking), HRMS's own fields, Pakistan NIC/NTN/STRN localization fields, CRM integration fields, and core system fields (`impersonate`). A genuinely new aimatic Custom Field/Property Setter/Client Script/Server Script is captured by `export-fixtures` automatically, no manual step. Only *newly appearing foreign/system* records ever need adding to `fixture_exclusions.json` — rare, and self-evident when it happens (an unexpected name shows up in an `export-fixtures` diff).
  - `Purchase Order`'s Items grid is styled by the `Client Script` **`PO Items Grid Enhancements (Claude)`** (enabled) — column widths/fullscreen/totals footer, deliberately built on top of Frappe's native `editable_grid` rather than reimplementing it. Other Client Script records on `Purchase Order` targeting the same grid (`POGRIDUI`, `PO Items Enterprise Grid (Claude)`) are disabled; keep only one such script enabled at a time.
  - **`Custom DocPerm` on `POS Opening Entry`/`POS Closing Entry`** (2026-07-15) — a plain filtered fixture (`[["parent", "in", ["POS Opening Entry", "POS Closing Entry"]]]`, no exclude-list needed since only aimatic's own rows exist there). Grants `POS User` create+submit on `POS Opening Entry`, and `POS Supervisor` create+submit+cancel on both — ERPNext's own core `DocPerm` for these two doctypes only grants **System Manager/Sales Manager/Administrator**, nothing at all for `POS User`/`POS Supervisor`, even though `offline_pos/api.py`'s own role model (`_ALLOWED_CASHIER_ROLES`/`_ALLOWED_CLOSE_SHIFT_ROLES`) clearly intends POS User to start shifts and POS Supervisor to close them. Without this, `start_pos_session`/`close_pos_session`'s `frappe.has_permission(...)` checks silently return `False` for a cashier who has exactly the roles the app's own UI/business logic says should work — confirmed on `siezal` with a real cashier account (`cashier1@aimatic.tech`, roles `POS User`+`POS Supervisor` only) before this fix.
- `after_install` runs `label_printing.setup.after_install` (creates label-printing roles/default templates) on a fresh `bench install-app aimatic`; the equivalent patches in `patches.txt` bring existing sites up to the same state via `bench migrate`. Print Format/Report packaging does **not** need an `after_install` entry at all — see `print_format` module-doc packaging below.

### Modules (`apps/aimatic/aimatic/`)

**`label_printing/`** — barcode/shelf label printing for retail (this is the module currently under active development).
- Doctypes: `AIM Label Template` (per-label-type layout: size, columns, fonts, which fields show, custom CSS, optional raw TSPL override) → `AIM Label Print Job` (a batch of labels to print, built from a submitted Purchase Receipt / Delivery Note / Sales Invoice / Material Transfer Stock Entry, or manually, holding `AIM Label Print Job Item` child rows) → print formats `aim_barcode_label` / `aim_shelf_label_a4`.
- `aim_label_print_job.py` `validate()` pipeline: `validate_template` (template must be enabled and match `label_type`) → `resolve_dates_and_defaults` (per-row MFG/expiry: manual override wins over system batch dates, via `resolve_final_date` in `utils.py`) → `validate_items` → `apply_totals` → `set_meta`.
- `utils.py` supplies shared helpers (`get_item_barcode`, `get_batch_dates`, `get_item_price`/`get_item_mrp`, `build_item_row`, `compute_job_totals`) also exposed to Jinja print formats via the `jinja.methods` hook (`get_barcode_data_uri`, `expand_print_rows`, `get_template_context`, `format_price`).
- `tspl.py` generates raw TSPL (TSC thermal printer language) from a job + template; `barcode_render.py` renders barcode data URIs (uses `python-barcode`, the app's one runtime dependency in `pyproject.toml`).
- `api.py` is the whitelisted surface used by the client scripts: `create_from_source_document` (generic across the 4 source doctypes, keyed by `_SOURCE_TYPE_CONFIG`/`_SOURCE_LINK_FIELDS`; `create_from_purchase_receipt` is now a thin back-compat wrapper over it), `get_items_from_source_document` (populates an existing/new Label Print Job's items when a source document is picked directly on that form, rather than via the source doc's own button), `query_source_documents` (shared Link-field query for all 4 source pickers: submitted docs only, latest first, supplier/customer or transfer warehouses shown as the dropdown description), `search_items`, `get_item_label_data`, `check_before_print`, `generate_tspl`, `mark_printed`.
- The source-document integrations are client-script-only (`public/js/purchase_receipt_label_printing.js`, `public/js/label_printing_source_buttons.js`); do not add server-side hooks on Purchase Receipt/Delivery Note/Stock Entry/Sales Invoice for this feature.

**`fbr_pos/`** — Pakistan FBR (Federal Board of Revenue) e-invoicing integration for POS Invoice. `payload_builder.py` builds the FBR submission payload from a POS Invoice (snapshotting item/tax/customer data onto `custom_fbr_*` fields), `api.py` submits it (sandbox/production URL and credentials come from an `FBR Integration Settings` doctype, keyed by company+branch, loaded via `settings.py:get_fbr_settings`), `tax_calculator.py` computes FBR-specific tax amounts per line, `accounting.py` adds/reconciles the resulting rows into the invoice's own Sales Taxes and Charges table (`add_fbr_sales_tax_row` — inclusive GST, backed into an effective rate via `On Net Total` so customer price doesn't change; `add_fbr_pos_fee_row` — an extra Rs. fee added on sales only, never on returns) and re-triggers `calculate_taxes_and_totals`, then aligns the cash payment row to the new grand total (`adjust_cash_payment_to_grand_total`, skipped for returns which carry explicit refund payments). Returns clear the copied `custom_fbr_*` snapshot fields (`events.py:clear_copied_fbr_fields_for_return`) so refunds are submitted to FBR independently rather than inheriting the original sale's FBR state.
- `Item.custom_fbr_tax_category` is mandatory in practice — `tax_calculator.py:get_item_fbr_configuration()` hard-throws at POS submit time if it's blank — so `patches.create_item_fbr_tax_rate_field` backfills any Item with no category to the `"Exempt goods"` `FBR Tax Category` record (its `company` field is never used to filter/scope lookups anywhere in `fbr_pos/`, so this is safe regardless of the Item's own company). The same patch adds a read-only `Item.custom_fbr_tax_rate` (`Percent`, `fetch_from: "custom_fbr_tax_category.tax_rate"`, inserted directly after the category field) so the applicable rate is visible on the Item itself without opening the linked `FBR Tax Category` doc; since `fetch_from` only populates on create/link-change, the patch also bulk-backfills it for every pre-existing category link.

**`offline_pos/`** — whitelisted API (`api.py`) backing both Electron POS and Android POS. Electron retains terminal-token authentication and a separate terminal-scoped cashier identity. Android never receives an API key/secret: a supervisor enrolls the device for one POS Profile, then each cashier authenticates with a Bearer OAuth2 session using Authorization Code + PKCE. `_is_bearer_authenticated_request()` is the hard compatibility boundary; do not derive Electron cashier identity from `frappe.session.user` or allow Android to fall back to terminal credentials.

- **Android device enrollment/auth, Mobile Sales, Customer Shopping, Restaurant Waiter Ordering** — four OAuth2 PKCE client surfaces (each its own public client, no client secret, identity always server-derived, never client-asserted). Android device enrollment/auth hooks live in the *nested* `aimatic/aimatic/offline_pos/` module (`device_events.py`/`device_auth.py` — note this is a different directory from the main `offline_pos/api.py`), while Mobile Sales/Shopping/Restaurant are their own top-level modules (`aimatic.mobile_sales.api`, `aimatic.shopping.api`, `aimatic.restaurant.api`). Refresh-token rotation/replay protection (`aimatic.aimatic.oauth.endpoints.get_token`) is shared infra across all four. See the **`oauth-client-surfaces`** skill for the full architecture, the client list, the idempotency records, and the client-side (Posapplication) entry points — and note that changing the POS Device hook path needs a site cache clear + Gunicorn reload, since Frappe caches `doc_events` in long-lived web workers.
- **Electron credential provisioning**: `provision_terminal_credentials` is an optional HTTPS-only first-run exchange for the desktop terminal path. It is rate-limited and audited; it must never be used by Android or Shopping.

- **Cashier auth** (`pos_cashier_login`): verifies a cashier's credentials against **Frappe's own password check** (`check_password` — never a manual hash compare) and role membership (`_ALLOWED_CASHIER_ROLES = {"POS User", "POS Supervisor", "System Manager"}`), scoped to one `terminal_id` + `pos_profile`. Returns capability flags (`can_refund`, `can_close_shift`, `can_void_items` — the last two derived from `_ALLOWED_CLOSE_SHIFT_ROLES`/`_ALLOWED_POS_ADMIN_ROLES` respectively, both currently `{"POS Supervisor", "System Manager"}`) and an offline-login validity window (`_CASHIER_OFFLINE_LOGIN_VALID_DAYS = 7`) so the terminal can let the cashier operate offline for up to a week. Every attempt (success or failure reason) is written to `POS Cashier Login Log` via `_audit_cashier_login` — password is never logged. `get_cashier_context` (Android Bearer analogue) returns the identical shape, including `can_void_items`, even though Android has no void-item UI yet — kept symmetric per its own docstring.
- **Admin/supervisor authorization** (`authorize_pos_admin_action` → `consume_pos_admin_authorization`): a two-step, HTTPS-only (`_require_https_for_pos_admin_authorization`), token-based step-up auth for sensitive terminal actions (`_ALLOWED_POS_ADMIN_ACTIONS = {"setup_pin", "reset_pin", "change_credentials", "close_shift", "void_item"}`). `authorize_pos_admin_action` checks a supervisor's password + role, then mints a random `secrets.token_urlsafe(32)` (only its SHA-256 hash is persisted, on `POS Admin Authorization`) with a 5-minute expiry. The actual single-use consumption logic (hash lookup, `SELECT ... FOR UPDATE` to prevent a double-consumption race, used/expiry checks, audit) lives in the private `_consume_pos_admin_authorization_token`; the whitelisted `consume_pos_admin_authorization` is now a thin wrapper over it. Every step is audited to `POS Admin Audit Log` via `_audit_pos_admin_action`.
  - **`close_shift`** (2026-07-16): lets a plain `POS User`'s own shift close once a *separate* `POS Supervisor` has authorized it, without the cashier logging out — `close_pos_session` gained an optional `supervisor_token` param; when `cashier_user`'s own roles don't satisfy `_ALLOWED_CLOSE_SHIFT_ROLES`, it calls `_consume_pos_admin_authorization_token` **in-process, in the same DB transaction as the close itself** (terminal_id derived server-side from the POS Profile's own `custom_terminal_id`, never trusted from the client). This is deliberate: if the close subsequently fails/rolls back for an unrelated reason, the token is never burned, since its consumption rolls back too. Before this, `close_pos_session` had no path at all for cashier-owns-shift/supervisor-authorizes separation of duties — it hard-required `cashier_user == opening.user` **and** that same account to itself hold the closing role.
  - **`void_item`** (2026-07-16, Electron client only, `Posapplication/src/renderer/renderer.ts`'s `voidCartRow`): gates F8/Delete/the cart-remove button/quantity-dropping-to-0 (all now funnel through this one function instead of three separate `cartLines.splice` call sites) behind the same primitive. Unlike `close_shift`, there is **no server document to bind the authorization to** — a cart is pure pre-sale client state — so the client calls `authorize_pos_admin_action` then *separately* calls `consume_pos_admin_authorization` itself before performing the local removal, rather than the server consuming it as part of some other action. This is real protection against normal use (a real supervisor password is checked against the server at the moment of voiding) but, stated plainly rather than assumed airtight, cannot stop a technically sophisticated user from bypassing the client-side JS entirely — there's no invoice yet to retroactively audit a bypassed void against, an inherent limit of any purely client-side cart control.
  - Both new actions are exercised end-to-end in `offline_pos/test_api.py` (`TestPosAdminAuthorization`, and `TestClosePosSessionCashier`'s token-based cases) — tests must patch only `frappe.local.request` (via `patch.object(frappe.local, "request", ..., create=True)`), never replace the whole `frappe.local` object the way `TestAndroidBearerSecurity` does for its narrower, single-attribute checks — `close_pos_session`/`authorize_pos_admin_action` touch far more of `frappe.local` (site, controller cache, etc.) mid-call, and wholesale replacement breaks those unrelated to the HTTPS check actually being tested.
  - The pre-existing `reset_pin`/`change_credentials` client flows (`requestSupervisorPinSetup`/`requestSettingsAuthorization` in `Posapplication`) mint a token via `authorize_pos_admin_action` but **never call `consume_pos_admin_authorization`** — only time-limited (5 min), not actually single-use in practice. Known, not fixed as part of adding `close_shift`/`void_item` (which do consume properly) — a candidate follow-up, not yet done.
- **Shift/session lifecycle**: a "session" is an ERPNext `POS Opening Entry`/`POS Closing Entry` pair. Electron's shared-terminal path looks it up by the explicitly authenticated terminal cashier, never the terminal service account. Android Bearer requests ignore client-supplied `cashier_user` and use `frappe.session.user`. `get_active_pos_session` finds the submitted, `Open`-status Opening Entry for `(pos_profile, cashier_user)`; if a different cashier has that profile's shift open, it is diagnostic only and is never silently reused. `start_pos_session` / `close_pos_session` / `get_pos_closing_summary` create/submit/report on those entries; `_require_refund_permission` gates refunds on the same shift.
- **Cart pricing** (`preview_cart`): builds an **unsaved** POS Invoice from a raw item list and returns priced/taxed rows + totals — never inserts, submits, calls the FBR API, touches coupon counts, or writes loyalty ledger entries. Still runs the real `fbr_pos` payload builder + accounting rows + `calculate_taxes_and_totals` so the preview matches what a real submit would produce. Returns both ERPNext totals and FBR-authoritative totals (`merchandise_total`, `value_excluding_tax`, `total_sales_tax`, `fbr_pos_service_fee`) — the client is expected to trust the FBR fields over its own math, and `line_amount` (not `net_amount`, which is post inclusive-tax-math) is the customer-facing price.
- **Sale/refund submission** (`submit_online_sale`, `submit_pos_refund`, `get_pos_invoice_for_refund`): actually inserts/submits POS Invoices (sale or return), idempotent on `terminal_invoice_id`/`terminal_refund_id` (`_find_existing_invoice`/`_find_existing_return` — a retried request returns the existing doc rather than double-submitting). Refunds re-validate requested quantities against what's already been returned (`_returned_qty_by_row`/`_validate_refund_quantities`) and preserve FBR snapshot values from the original row (`_preserve_original_return_row_values`) rather than recalculating them.
- Shared builder: `_build_pos_invoice_doc` is the one place that assembles a POS Invoice from `(pos_profile, customer, items, ...)` — used by both `preview_cart` and the real submit paths, so pricing/tax logic can't drift between preview and submit. It sets `update_stock` from the POS Profile (`cint(pos.get("update_stock"))`) on the initial doc dict, since every real POS Profile has `update_stock=1` configured (`POS Profile.update_stock` is itself `hidden`/`read_only`/`default="1"` in ERPNext).
- **POS Invoice stock/GL updates only ever happen via the *consolidated* Sales Invoice, never the individual POS Invoice** — `POSInvoice.on_submit()` (`erpnext/accounts/doctype/pos_invoice/pos_invoice.py`) does **not** call `super().on_submit()` (its `validate`/`on_cancel` do call `super(SalesInvoice, self)...`, but `on_submit` doesn't), so `SalesInvoice.on_submit()`'s `update_stock_ledger()`/`make_gl_entries()` never run for a POS Invoice directly — regardless of its own `update_stock` value. Stock/GL only move when a `POS Closing Entry` is submitted (`close_pos_session`) → `on_submit` calls `consolidate_pos_invoices()` → `POS Invoice Merge Log.merge_pos_invoice_into()` builds a real `Sales Invoice` via `frappe.model.mapper.map_doc(pos_invoice, sales_invoice, ...)`, which copies same-fieldname values including `update_stock` (no `no_copy` flag on that field) — and it's *that* Sales Invoice's own `on_submit()` that actually calls `update_stock_ledger()`. Net effect: a correctly-flagged POS Invoice sitting in an un-closed shift shows no stock movement yet, by design — that's expected, not a bug, until the shift is closed.
- **Historical stock backfill (szl only)**: `_HISTORICAL_POS_STOCK_CORRECTION_ENTRIES` in `vendor_performance/api.py` hardcodes the names of 10 one-off `Stock Entry` (Material Issue) documents that backfilled stock for POS Invoices submitted before the `update_stock` fix above — added as an explicit second match arm in `_get_cogs_summary`/`_get_cogs_by_item`'s `voucher_type` filter (deliberately a named, bounded exception, not a general "count all Stock Entries" rule, which would wrongly pull in unrelated write-offs). **Known gap**: `_get_cogs_by_voucher` (per-transaction COGS in `get_vendor_recent_sales`'s "last 3 sales" list) can't use the same fix — these backfill entries are netted per item/date across multiple invoices with no 1:1 mapping back to a single POS Invoice, so sales from that historical window still show `cogs_amount: 0` in that one drill-through view.
- Audit/session doctypes live in `aimatic/aimatic/doctype/`: `POS Cashier Login Log`, `POS Admin Audit Log`, `POS Admin Authorization` — all currently thin `Document` subclasses (no custom `.py` logic beyond what's shown above; the logic lives in `offline_pos/api.py`, not on the doctype controllers).
- `customer_validation.py` (hooked on `Customer.validate`) normalizes Pakistani mobile numbers to `+92XXXXXXXXXX` form (`normalize_pak_mobile` — handles `0XXXXXXXXXX`, `92XXXXXXXXXX`, `3XXXXXXXXX`, and `00`-prefixed international dialling) and enforces one Customer per normalized number via a direct equality lookup (safe only because normalization runs on every save, so all persisted values are already canonical).

**`branch_management/`** — enforces a strict Branch → Cost Center → Warehouse model so normal Sales/Purchase/Stock users never pick (and can't mis-pick) cost center/warehouse themselves; derived entirely from the user's assigned Branch (assigned via a `User Permission` row, `allow: Branch`, `is_default: 1` — that `is_default` flag is easy to miss when assigning by hand in the Desk UI and, without it, `apply_branch_defaults` silently sees no default branch at all). `events.py`: `apply_branch_defaults` (`before_validate` hook, shared across Sales Order/Sales Invoice/Delivery Note/Purchase Order/Purchase Invoice/Purchase Receipt/Stock Entry) fills header-level `cost_center`/`set_warehouse`/`rejected_warehouse` from `get_branch_defaults()`, then `_apply_item_row_defaults` forces the same onto every item row directly (header values don't reliably cascade to rows through ERPNext's own mechanisms — see the hook-ordering note above); rows with no `rejected_qty` always get `rejected_warehouse` cleared to `None`, regardless of override status, since a stray value there is never meaningful. Branch/Sales/Purchase/Stock/Accounts Managers and System Managers (`user_can_override`) may still override — for them it only fills blanks, never overwrites a deliberate choice. POS Invoice (including `is_pos` Sales Invoices) is skipped — it's built server-side from POS Profile, which already carries cost_center/warehouse. `validate_branch_company_consistency` (Branch validate hook) re-checks that a Branch's cost_center/finished_goods_warehouse/rejected_warehouse still belong to the Branch's own Company whenever Company changes (link_filters only filter the picker, they don't re-validate an already-set value). `sync_branch_to_user` (`User Permission` `after_insert`/`after_delete` hook) mirrors a user's default Branch onto `User.custom_branch`, a read-only display-only field (`in_list_view: 1`) so it's visible as a column on the User list — not the source of truth, just a mirror.
  - **`validate_pos_profile_cost_center`** (`POS Profile` validate hook, 2026-07-15) — throws if `POS Profile.cost_center` resolves to a group (`is_group=1`) Cost Center. POS Invoice is deliberately outside `apply_branch_defaults`'s scope (see above — it's built server-side straight from POS Profile), so nothing else in the app was checking this field. Real incident: `hsm`'s `POS Profile HSM-002` was pointed at the company's auto-created root Cost Center (`Hatim Super Market - HSM`, a group) instead of the leaf `Main - HSM` one level under it. ERPNext only rejects a group cost center at GL-entry time, which for a POS Profile means shift-close, deep inside `POSClosingEntry.on_submit → consolidate_pos_invoices → create_merge_logs`; worse, ERPNext's own exception handler there reacts to that failure with a `frappe.db.rollback()` that wipes out the just-inserted POS Closing Entry row, then tries to mark that now-nonexistent document `"Failed"` via `add_comment()`, which itself fails link validation — so the error actually surfaced to the client was a misleading `LinkValidationError: Could not find Reference Name: POS-CLO-...`, not the real Cost Center error. This hook catches the real problem at POS Profile save time instead.
  - **`Warehouse.custom_branch`** (new custom field, patch `create_warehouse_branch_field`, 2026-07-11) is the reverse lookup Branch itself doesn't provide — `Branch.finished_goods_warehouse`/`rejected_warehouse` only point forward to exactly 2 warehouses, with nothing pointing back, so "all warehouses for branch X" wasn't queryable before this. Patch-backfilled from those same 2 fields on every existing Branch (idempotent, only fills blanks). `get_branch_defaults()` in `utils.py` still only returns those same 2 fixed fields for its own doc-defaulting purpose — don't reuse it for anything needing the general "all warehouses for this branch" answer (e.g. `vendor_performance`'s `_get_branch_warehouses`, see below); query `Warehouse.custom_branch` directly instead, since a branch may have more than 2 warehouses in the future.
  - **Stock Entry per-row `branch` derivation** (2026-07-12) — `Stock Entry Detail.branch` is an Accounting Dimension field (auto-injected onto every doctype the dimension is registered against, including child tables; not something `branch_management` defines), mandatory for P&L accounts, but nothing populated it per row, so every GL line silently inherited the single parent-level `branch` — wrong for inter-branch Material Transfers, and left blank (a late GL-entry error at submit, not save) for override users with no default Branch. Fixed via `utils.get_warehouse_branch()` (reverse lookup through `Warehouse.custom_branch`, may be `None` for unmapped/generic/disabled warehouses): `_apply_item_row_defaults` now sets each row's `branch` from its own `s_warehouse`/`t_warehouse` (falling back to `doc.branch`) whenever the row's `branch` is blank, regardless of override status — this is a derived fact, not a user choice, so there's nothing to "override". Separately, `apply_branch_defaults` calls `_infer_stock_entry_branch(doc)` to derive a document-level `branch` from row warehouses when an override user has none of their own, before falling through to the existing "no branch, can't proceed" logic. `Stock Entry Detail.branch` is also a fixture-tracked `read_only` Property Setter (2026-07-12) — left editable it would sit ambiguously next to a row's own Source/Target Warehouse with no indication which one it reflects, confusing for users who only think in terms of Branch (not Cost Center) to begin with; read-only keeps it purely code-derived, consistent with how `cost_center`/`warehouse` are already locked down elsewhere in this module.
  - **Branch-restricted users need User Permission on Cost Center and Warehouse too, not just Branch** — a User Permission on one Link doctype (e.g. Branch) does nothing to restrict a completely different Link doctype (Cost Center, Warehouse) even though they're conceptually related; Frappe has no cascading between them. Without explicit Cost Center/Warehouse User Permissions, a branch-restricted non-override user still sees every branch's cost centers/warehouses in dropdowns (though `apply_branch_defaults` will still force-correct the actual saved value for non-override users). A user who ever handles Purchase Receipt (or anything with a rejected-quantity flow) needs **two** Warehouse permissions — their branch's accepted warehouse *and* its own per-branch Rejected Warehouse — not just one; a stale/wrong second permission (e.g. still pointing at the old generic `Rejected Warehouse - ST`) doesn't itself break anything server-side, but is worth cleaning up per-user when branches or warehouses are restructured.
  - If a field's value looks like it's copying the wrong thing (e.g. a Link's own name instead of one of its own fields), check `frappe.get_meta(doctype).get_field(fieldname).fetch_from` — `Purchase Receipt Item.cost_center` was once misconfigured this way (`"branch.branch"` instead of `"branch.cost_center"`).

**`item_pricing/`** — `events.py`'s `update_latest_price_incl_taxes` (Purchase Invoice/Purchase Receipt `on_submit`) pushes each submitted line's `custom_price_after_taxes` onto both the `Item` master (`custom_latest_price_incl_taxes` + `custom_latest_price_source_date`) and every existing `Item Price` row for that item — done as a direct write rather than relying on `Item Price`'s `fetch_from`, since `fetch_from` only copies once (on create/item_code change), not live. A more recent `custom_latest_price_source_date` always wins, so a backdated/late submission can't clobber a newer price.

**`purchase_history_autofill/`** (Purchase Receipt only, added 2026-07-17) — auto-fills empty/zero item-level purchase fields (GST %, Advance Tax %, FED %, their `Tax Formula` Link fields, Trade Offer, Discount %, MRP, Shelf Price) from the most recent submitted Purchase Receipt/Purchase Invoice row for the same Supplier + Branch + Item, whether the row arrived via "Get Items From Purchase Order" or was added manually. Deliberately narrower in scope than it could be — see the explicit exclusions below, all enforced structurally in `utils.py`'s `_ALWAYS_EXCLUDE`, not just by the "only if empty" check:
- **Never touches Quantity, Rate, or `custom_vendor_rate`** (the field the client-side tax-calc scripts — `prv1` on Purchase Receipt, `PurchaseInvoice` on Purchase Invoice — treat as the actual rate input; `rate`/`price_list_rate` are themselves Property-Setter `read_only=1` on Purchase Receipt Item, computed from `custom_vendor_rate` + the tax fields this module *does* fill).
- **Never touches the read-only computed-output custom fields** (`custom_price_after_taxes`, `custom_gst_amount`, `custom_advance_tax_amount`, `custom_fed_amount`, `custom_gross_total`, `custom_line_total`, `custom_base_cost_price`, `custom_trade_offer_total`, `custom_old_price_after_tax`, ...) — those are recalculated client-side from the *input* fields this module populates (percentages, formulas, MRP, Trade Offer), so populating the outputs too would be immediately overwritten or, worse, briefly show a stale number before that recalculation runs.
- **Field discovery is schema-driven** (`get_autofillable_fields`, via `frappe.get_meta` — never hardcoded per-site), so it adapts automatically if a site's custom field set differs; deliberately **does not** add its own process-level cache on top of `frappe.get_meta`, since szl/hsm/siezal share one gunicorn worker process and a naive `functools.lru_cache` keyed only by doctype name would leak one site's field set into another site's request — `frappe.get_meta` already caches correctly per-site on its own.
- **Matching is row-level, not document-level**: `fetch_latest_history_rows` queries `Purchase Receipt Item`/`Purchase Invoice Item` directly (`docstatus=1 AND is_return=0 AND supplier=X AND branch=Y AND item_code IN (...)`), batched across every item on the current document in one query per source doctype (not N+1), and picks the single most-recent row per item (`posting_date`, `posting_time`, `creation`, `name`, `idx`, all descending — same tie-break convention as `purchase_printing.populate_old_purchase_snapshot`'s existing history lookup). Because the query is row-level, an item missing from the supplier+branch's most recent document naturally falls through to an older document that does contain it, with no extra "walk backwards through documents" logic needed.
- Hooked as a third `before_validate` entry on Purchase Receipt (after `apply_branch_defaults`, so `doc.branch` is already resolved, and after `populate_old_purchase_snapshot`) — fires identically for both "Get Items From Purchase Order" and manually-added rows, since both reach `before_validate` the same way at save time.
- `api.py`'s `preview_item_history` + `public/js/purchase_receipt_history_autofill.js` (second entry in `doctype_js["Purchase Receipt"]`, a list) are an optional live pre-save convenience only — they pre-fill blank grid cells on `item_code`/`supplier`/`branch` change so the receiving user sees the carried-over values before hitting Save. The `before_validate` hook is what actually guarantees the fields get populated; the client script could be removed without breaking that guarantee.
- No new custom fields or doctypes were needed — every field this module reads/writes already existed identically on szl/hsm/siezal (confirmed 88 matching Custom Field rows across `Purchase Order Item`/`Purchase Receipt Item`/`Purchase Invoice Item` on all three sites before building this), so shipping it is a pure code deploy (commit + restart), no `bench migrate` required.

**`sales_dashboard/`** (added 2026-07-16) — real-time POS sales dashboard for branch/store managers and leadership, at its own route `sales-dashboard-console` (`aimatic/aimatic/page/sales_dashboard_console/`, same route-collision-avoidance reasoning as `vendor-performance-console` above), reachable via the root `Aimatic` workspace's "Sales Dashboard" shortcut (a direct root-level shortcut, not a dedicated child workspace like Vendor Performance/Label Printing get — deliberate, since this page has no companion doctypes to link to).
- **Data source**: every query in `aimatic.sales_dashboard.api` sums `POS Invoice.grand_total` directly for `docstatus=1` rows (never `POS Closing Entry`) — `grand_total` is fully computed (tax, FBR fee) at submit time, so this is accurate without waiting for shift close, and `is_return=1` rows carry a negative `grand_total` so a plain `SUM` nets returns automatically. All three sites currently have `POS Settings.invoice_type == "POS Invoice"`; if that's ever switched to `"Sales Invoice"` on a site, every query in this module needs updating too or the dashboard silently goes blind for that site.
- **Live open-shift totals**: `_compute_live_shift_total`/`_get_open_shifts` replicate ERPNext's own `pos_closing_entry.build_invoice_query` technique (`owner == user`, `is_pos == 1`, `pos_profile` match, submitted invoices between the `POS Opening Entry`'s `period_start_date` and now) so a shift that hasn't been closed yet still contributes an accurate running total to both the "Active Shifts" KPI and that profile's card — a still-open shift is never invisible just because no `POS Closing Entry` exists.
- **Two distinct per-profile numbers, not one**: each POS Profile card's headline "today's sales" is the full-day total across every shift/cashier that used that profile today (`posting_date = today`); when a shift is currently open, a separate smaller "Current shift" sub-line shows only that specific shift's live running total. These can legitimately differ (shift change mid-day, a reopened shift) — they are deliberately never conflated into one number.
- **Today for cards, date-range for KPIs/trend**: the branch → POS Profile card grid always shows "today" regardless of the toolbar's date-range filter (Today/Yesterday/Last 7 Days/Last 30 Days/This Month/Last Month/Custom), matching the literal "current day sales" ask; the top KPI row and Branch Comparison chart honor the date-range filter; the Sales Trend chart is a separate fixed 14-day rolling window (`_TREND_DAYS`), independent of the date-range filter, since a single-day preset would otherwise render as a one-point "trend."
- **Branch scoping and the Unassigned bucket**: `_resolve_branch_filter` returns `None` (no SQL filter — full company total, including branch-less rows) unless the caller explicitly picks one branch, or the current user's own Branch User Permissions grant visibility into a strict subset of the company's branches (detected by comparing permission-filtered vs. unfiltered `Branch` counts) — in which case it restricts to that visible set and Unassigned rows are excluded too, a deliberately conservative default for a restricted user. Every branch-grouping query resolves branch via `COALESCE(pi.branch, pp.branch)` (a POS Invoice's own `branch` first, falling back to its POS Profile's `branch`) since at least one real POS Profile (siezal's `Food Panda`) has a blank `branch` — those rows bucket into a synthetic "Unassigned" branch section, shown only when nothing has already narrowed the scope to specific branches. Enabled (`disabled=0`) POS Profiles only — a disabled profile would otherwise show as a permanent zero-value ghost card.
- **UI**: `sales_dashboard_console.js`/`.css` follow `vendor_performance_console`'s KPI-card visual taxonomy (`sd-accent-*` classes, same Frappe theme CSS variables for light/dark correctness) but use `frappe.ui.Dialog` popups for drill-through instead of `vendor_performance_console`'s inline expand-in-place sections — chosen so N branches × M always-visible profile cards don't turn the page into an ever-growing scroll. Charts (Sales Trend line, Branch Comparison bar, per-profile Hourly Breakdown bar) use the framework's bundled `frappe.Chart` (Frappe Charts) — no new frontend dependency. Auto-refreshes every 60 seconds (pause/resume via the page's secondary action) plus a manual "Refresh" primary action, matching `vendor_performance_console`'s manual-refresh convention.
- **Access**: gated by role (`System Manager`/`Sales Manager`/`Accounts Manager`/`POS Supervisor`) inside the API layer, not the Page's own `roles` field (`roles: []`, same convention as `vendor_performance_console`) — enforcing at the API means a direct whitelisted-method call is checked the same way the UI is.
- **Known, accepted caveat**: like `vendor_performance`'s own note above, `POS Invoice.branch` on an older invoice reflects the POS Profile's branch *at invoice time*, which can drift from the profile's branch today if it was ever reassigned — low risk at the current small transaction volume, not fixed here.

### Desk navigation (`Workspace` + `Workspace Sidebar`)

Standalone doctypes get Desk navigation via a `Workspace` "Aimatic" (`aimatic/aimatic/workspace/aimatic/aimatic.json` — the home-grid tile: header, shortcuts, then `card`/`Card Break` groups covering **POS & Devices**, **Loyalty & Gift Vouchers**, **FBR / Tax**, **Pricing**, **Restaurant**, **Shopping**, **Mobile Sales**) **and** a separate `Workspace Sidebar` "Aimatic"/"Label Printing" (`aimatic/workspace_sidebar/*.json` — the actual left-nav tree; a `Workspace` alone does not populate it in this Frappe version). See the **`desk-navigation`** skill for the sync mechanics, the `parent_page` top-level-tile rule (used to fold `Vendor Performance`/`Label Printing` under Aimatic), and the same-slug Workspace-vs-Page route collision. Custom Page routes so far: `vendor-performance-console`, `sales-dashboard-console`, `ai-assistant-console` — all `-console`-suffixed to avoid colliding with an existing or future same-slug Workspace, all reached via a root `Aimatic` workspace shortcut.

- **Summary-first, drill-through UI** — `aimatic.vendor_performance.api.py` is deliberately split into one cheap always-loaded summary endpoint plus several on-demand detail endpoints, not one monolithic call, because the full per-item/per-transaction breakdown doesn't scale to vendors with heavy transaction history. `get_vendor_performance_summary` (called on every filter change) returns only aggregate `SUM`/`COUNT` numbers — supplier-SKU exposure stock (from `Bin`, already-materialized current stock), sales/purchase/outstanding totals, last payment — no per-item rows and no stock-ledger replay. Everything else is a separate whitelisted call the page JS only fires when the user clicks a section's "Load" button: `get_vendor_stock_detail` (current stock of supplier-linked SKUs, top N by value), `get_vendor_origin_stock_detail` (the FIFO-replay exact vendor-origin stock, the most expensive computation — see guard below), `get_vendor_recent_purchases`, `get_vendor_recent_sales`. Each drill-through section resets to its unloaded placeholder state whenever the filters (supplier/company/days) change, so stale data for a different vendor is never shown.
- **Origin-stock replay guard** — `get_vendor_origin_stock_detail` replays every `Stock Ledger Entry` for the supplier's linked items to attribute current stock back to its purchase origin (phase-2 exact vendor-origin stock), an O(all-time ledger rows) Python computation. Before running it, it does a cheap `COUNT(*)` first; past `_MAX_ORIGIN_STOCK_SLE_ROWS` (150,000) it returns `{too_large: True, sle_row_count: N}` instead of attempting the replay, and the page shows a message instead of hanging. Raise this constant only if the replay is confirmed fast enough in practice for that row count.
- Shows: exact vendor-origin stock (phase-2 FIFO attribution, drill-through), supplier-SKU exposure stock (drill-through), sales revenue in a configurable recent window alongside **Cost of Goods Sold and Gross Margin** (COGS is cost-basis value from Stock Ledger Entry valuation for supplier-linked SKUs consumed by submitted sales, not retail revenue — `_get_cogs_summary`/`_get_cogs_by_item`/`_get_cogs_by_voucher` sum `-stock_value_difference` from SLE rows where `voucher_type IN ('Sales Invoice', 'POS Invoice')`), **Purchase Invoices and Purchase Receipts as two separate summary cards** (`_get_purchase_summary` vs `_get_purchase_receipt_summary` — kept distinct rather than folded together since billed amount and received cost basis can genuinely diverge: received-not-yet-billed, or billed items with no physical receipt), outstanding payable, last payment (all in the summary), last 3 purchase invoices, and last 3 sales (drill-through, each showing per-transaction COGS/gross margin too).
- **`_get_sales_summary`/`_get_sales_by_item`/`_get_recent_sales` are POS Invoice only** (fixed 2026-07-17, previously unioned with Sales Invoice Item/Sales Invoice) — see the **"Revenue double-counting gotcha"** note under the `ai/` module above for why: every submitted Sales Invoice on this bench is itself a POS-consolidation output (`consolidated_invoice`/`consolidated_credit_note`), so unioning it back in double-counted every closed-shift sale's revenue, inflating this page's sales/gross-margin numbers. Do not reintroduce a Sales Invoice union to these three functions without re-checking that assumption still holds (i.e. that no genuine standalone Sales Invoice workflow has been added since).
- **Visual design** — the page (`vendor_performance_console.css`/`.js`) uses a color-coded KPI card grid (icon chip + accent border per category: stock/revenue/cogs/margin/purchase/receipt/payable/payment), right-aligned tabular numbers, red/green gross-margin coloring, and polished empty/loading states, all built on Frappe's own theme CSS variables so it stays correct in light/dark mode. Numeric formatting in JS must go through `format_number()`/`money()`/`percent()` in this file, never raw `frappe.format(value, {fieldtype: 'Float'/'Currency'})` — that helper wraps output in a `<div style="text-align: right">` meant for grid cells, which shows up as literal escaped HTML text when concatenated into a card's subvalue string.
- **Abnormal Sell-Through drill-through** (`get_vendor_abnormal_ratios`) — a shrinkage/theft-adjacent signal, deliberately *not* a Purchase Order-vs-Purchase Receipt document comparison. Rationale (per explicit product decision): a vendor who physically under-delivers while the Purchase Receipt still records the agreed quantity (a deceived or colluding receiving count) won't show up as a PO-vs-PR mismatch, since both documents agree with each other — the deception happens below what any document captures. It only surfaces later as fewer units than expected actually being available to sell. A flat purchased:sold ratio over a fixed window can't distinguish "80% sold in 2 days" (fine, way ahead of any pace) from "80% sold in 30 days" (not fine, if the item normally clears faster) — both look identical with no time context. So per item: `_get_purchase_events_by_item` finds purchased qty in-window plus a qty-weighted average arrival date (not per-Purchase-Receipt cohort tracking); `_get_item_velocity_baseline` establishes that item's own normal daily sell-through pace from a 180-day trailing window (`_VELOCITY_BASELINE_DAYS`) that ends *before* the evaluation window starts (excluding it deliberately — otherwise a genuinely underselling delivery drags its own baseline down and looks normal against itself, especially visible for items with little sales history predating the window); `expected_days_to_clear = purchased_qty / daily_rate` is compared against actual elapsed days — still-early is never flagged regardless of how little has sold, past-expected with a low sold percentage escalates through `normal`/`abnormal`/`critical` severity. Items with too little baseline sales history (`< _VELOCITY_MIN_BASELINE_SOLD_QTY`) fall back to comparing their plain in-window ratio against the vendor's *other* fallback-method items via Tukey's IQR fence (`_percentile`, a small dependency-free implementation — no numpy/scipy added) or a fixed sanity multiple with too few items for that to be meaningful. Rendered with severity-colored rows (`vp-row-flagged` red / `vp-row-warning` yellow) and `vp-badge-danger`/`vp-badge-warning` status badges.
- **Optional Branch/Warehouse scoping** (all 6 endpoints, 2026-07-11) — every metric/drill-through can additionally be scoped to a Branch and/or Warehouse, both optional; blank behaves exactly as before. Resolved once via `_resolve_branch_warehouse_scope(branch, warehouse)`: a specific Warehouse always wins over a simultaneously-set Branch (its own `custom_branch` is derived and used for header-level filtering, never validated against a mismatched Branch param) so the UI can't produce a contradictory combination. `Sales Invoice`/`POS Invoice`/`Purchase Invoice`/`Purchase Receipt`/`Payment Entry` headers carry `branch` directly; `Stock Ledger Entry`/`Bin` only carry `warehouse` (no `branch` column, ever), so a selected Branch is expanded to its warehouse list (`_get_branch_warehouses`) for those. If a Branch resolves to zero tagged warehouses, stock/COGS metrics fall back to unfiltered rather than a `WHERE warehouse IN ()` that would zero everything out. `_get_supplier_item_codes` (the item universe) is deliberately left unscoped — branch/warehouse narrow the *metrics* per item (correctly showing `0`), not which items appear at all. `_get_item_velocity_baseline`/`_get_sold_qty_since` (used by the abnormal-ratios check) are scoped by the same warehouse list as everything else in that check for an apples-to-apples comparison. `_get_exact_vendor_origin_stock`'s FIFO replay has a known, accepted limitation when warehouse-scoped: stock that arrived via an inter-branch Material Transfer (still explicitly allowed) shows as unattributed at the destination warehouse rather than correctly tracing back to the original supplier, since a transfer's incoming SLE row has no `voucher_detail_no` link back to a Purchase Receipt/Invoice row. `frappe.has_permission('Branch', ...)` gates a resolved branch the same way Supplier is already gated in `_resolve_context`.
- **Data-quality caveat, not a code bug**: `branch` is inconsistently populated on older POS Invoices/Purchase Receipts predating this filter (`apply_branch_defaults` explicitly skips POS Invoice, per `branch_management`'s note above). A Branch/Warehouse-scoped query on header-level metrics (sales, purchases) will therefore under-count older activity that never got a `branch` value, while stock/COGS metrics (warehouse-based, from `Stock Ledger Entry`) aren't affected since warehouse consolidation was thorough. Worth a backfill if this becomes operationally important.

- **FBR payment-mode misreporting with large Gift Voucher redemptions**: `fbr_pos/payload_builder.py:get_payment_mode()` picks the e-invoicing `PaymentMode` field by whichever payment row on the invoice has the largest amount, then matches "cash"/"card" in its Mode of Payment name, else falls back to `FBR Integration Settings.default_payment_mode`. Gift Voucher redemption is its own Mode of Payment row (see `gift_voucher/` above); if a voucher redemption is the largest row on a sale (e.g. a big voucher covering most of the bill), the FBR submission reports the fallback default payment mode instead of the actual cash/card portion actually paid. Not a crash, just a misreported field on e-invoicing submissions for that subset of sales — flagged, not fixed.

## Branch ↔ Warehouse ↔ Cost Center model (no generic fallback)

Per explicit product decision, there is no generic/shared warehouse or cost center anywhere in this system — every stock movement and GL entry lands in a branch-specific warehouse/cost-center, enforced in code by `branch_management/apply_branch_defaults` (see above), with inter-branch Material Transfers still allowed. The old generic fallbacks have been fully retired on both sites (szl's `Test Company`, szl's `Siezal Super Market`, and production `siezal`):

- `Stores - ST`/`Stores - SSM` (generic default warehouse) and `Main - ST`/`Main - SSM` (generic default cost center) are **disabled** everywhere, along with the unused-for-retail `Finished Goods`/`Work In Progress`/`Goods In Transit` warehouses (no manufacturing happens in this business; these were dead ERPNext defaults with zero stock/references).
- **Rejected Warehouse was also generic** (one shared `Rejected Warehouse - ST`/`-SSM` used by *every* branch via `Branch.rejected_warehouse`) — this is the same class of bug as `Stores`/`Main` and was fixed the same way: a dedicated `Rejected <Branch> - ST`/`-SSM` warehouse now exists per branch, with each `Branch.rejected_warehouse` repointed at its own.
- `Item Default.default_warehouse`, `Stock Settings.default_warehouse`, and `Company.cost_center`/`round_off_cost_center`/`depreciation_cost_center` no longer reference any generic record on either site.
- `Purchase Taxes and Charges Template` masters (e.g. `Standard Rate Goods (Default) - ST`, `Pakistan Tax - SSM`) had `Main`/`Main - SSM` baked in as their default tax-row cost center — also repointed, since new tax rows built from these templates would otherwise silently reintroduce the generic cost center.
- szl is a test bench for two companies (`Test Company`, real branches/data; `Siezal Super Market`, an empty template stub with zero branches — not yet actively used) — since `Test Company` has no reliable per-branch attribution for its pre-existing data, all of its existing stock/drafts were consolidated onto one branch, **Bahria Town Phase VIII Rwp**, as a deliberate simplification (szl is for testing workflows before they're finalized on production `siezal`, so exact historical branch attribution there doesn't matter). Production `siezal` has one real branch, Ghouri Town Phase V, and its company-level defaults now point at that branch's own cost center.
- **Suggestion for when siezal adds its next branches** (currently has 1 of a planned 6): consider a dedicated non-branch "Head Office"/corporate Cost Center for `Company.round_off_cost_center`/`depreciation_cost_center`, rather than pointing those at one specific branch's cost center — mixing head-office rounding/depreciation postings into a single branch's cost center will pollute that branch's own P&L once there's more than one branch to compare against. Not done yet (only one real branch exists today, so it's moot for now); revisit before/when the next siezal branch goes live.
- **Operational gotchas learned while doing this migration** (worth knowing before touching Warehouse/Cost Center records again):
  - GL Entry's `cost_center` is driven by each **row's own** `cost_center` field (e.g. `Stock Entry Detail.cost_center`), not the parent document's — ERPNext falls back to the parent value only when the row's own field is blank (`accounts_controller.py:get_gl_dict`). Auto-defaulting (e.g. from `Company.cost_center` at insert time) can silently bake in whatever the company default was *at that moment*, so always set row-level cost_center/warehouse explicitly when scripting transactions rather than trusting auto-default, especially mid-migration when defaults are actively changing.
  - `Branch` (the Accounting Dimension) was *not* auto-populated per-row on Stock Entry the way `cost_center` is — every GL line from a Stock Entry inherited the single parent-level `branch` value, wrong for inter-branch Material Transfers specifically (the "from" and "to" legs got the same Branch dimension tag even though they touch two different branches' warehouses). **Fixed 2026-07-12** — see `branch_management`'s "Stock Entry per-row `branch` derivation" note above (row-level branch is now derived from each row's own warehouse via `Warehouse.custom_branch`).
  - Disabling a Warehouse/Cost Center blocks not just *new* transactions but also **cancelling old ones that reference it** (`erpnext/stock/utils.py:validate_disabled_warehouse` fires on the reversal Stock Ledger Entry a cancel creates) — if a historical document needs to be cancelled/amended after cleanup, temporarily re-enable the record, fix it, then re-disable.
